✅ Tus definiciones de los 7 conceptos.

¿Qué es un dato?
Un dato es una representación simbolica de una variable cualitativa o cuantitativa que por si solo no trasmite información útil, para ello tiene que ser organizada y complementado con más datos.

¿Qué es información?
es un conjunto de datos organizados que trasmite un mensaje útil.

¿Qué es inteligencia de negocio (Business Intelligence)?
Es un conjunto de tecnologías, aplicaciones utilizadas para recopilar, analizar y transformar datos en información visual y valiosa.

¿Qué es un KPI?
es un indicador clave de rendimiento, son metricas cuantificables que evaluan el éxito y el progreso de un negocio

¿Qué diferencia hay entre una base de datos relacional y un archivo Excel?
Una base de datos relacional está estructurado de forma ordenada y se maneja por tipo de datos es escalable, en cambio un archivo excel tiene un limite de filas, y es una hoja de calculo lo cual no necesariamente cada columna tienen un mismo tipo de datos.

¿Qué hace realmente un Data Analyst?
El analista de datos es el puente entre datos sin procesar e información valiosa.
El analista de datos recolecta, limpia y analiza grandes cantidades de datos, identifica patrones y forumula tendecias, transformando datos en información valiosa que sirven para la toma de deicisiones

¿Cuál es la diferencia entre Data Analyst, Data Engineer, Data Scientist y Business Analyst?

Data Enginner: Construye la ingraestructura
Bussiness Analyst: se enfoca en el negocio
Data Analyst: extrae conclusiones del pasado
Data Scientist: crea modelos para predicir el futuro

Un error muy común es aprender Power BI, SQL o Python sin entender el flujo completo de análisis de datos. Eso provoca que muchas personas sepan usar herramientas, pero no resolver problemas de negocio.

¿Qué hace un Data Analyst?

Imagina que el Gerente General de una agroexportadora te dice:

"Las ventas aumentaron, pero las ganancias disminuyeron. ¿Qué está pasando?"

No espera que le muestres una consulta SQL.

Espera una respuesta.

Para responderla, un Data Analyst sigue un proceso estructurado.

Paso 1. Entender el problema

Nunca abras Excel primero.

Primero debes hacer preguntas.

Ejemplo:

•	¿Qué periodo analizamos? 

•	¿Qué productos? 

•	¿Qué sedes? 

•	¿Qué significa "ganancia"? 

•	¿Existe presupuesto esperado? 

•	¿Qué decisiones se tomarán con el análisis? 

Un analista dedica gran parte del tiempo a comprender el problema antes de tocar los datos.

Paso 2. Obtener los datos

Los datos pueden venir de:

•	Excel 

•	ERP (SAP, Oracle) 

•	SQL Server 

•	PostgreSQL 

•	APIs 

•	Archivos CSV 

•	Google Sheets 

•	Sistemas internos 

•	Formularios 

Un buen analista sabe conectar todas estas fuentes.

Paso 3. Limpiar los datos

Aquí suele invertirse el mayor esfuerzo.

Tareas comunes:

•	eliminar duplicados 

•	corregir errores 

•	completar valores faltantes 

•	convertir formatos de fecha 

•	unificar nombres 

•	validar consistencia 

En proyectos reales, esto puede representar entre el 60 % y el 80 % del tiempo.

Paso 4. Analizar

Aquí comienzan las preguntas importantes:

•	¿Qué ocurrió? 

•	¿Por qué ocurrió? 

•	¿Dónde ocurrió? 

•	¿Cuándo ocurrió? 

•	¿Quién estuvo involucrado? 

•	¿Qué tendencias existen? 

•	¿Qué variables parecen influir?

Paso 5. Visualizar

No basta con tener respuestas.

Hay que comunicarlas de forma clara mediante:

•	gráficos 

•	tablas 

•	indicadores 

•	dashboards 

Paso 6. Contar una historia (Storytelling)

No es suficiente decir:

"Las ventas bajaron un 15 %."

Un analista aporta contexto:

"Las ventas disminuyeron un 15 % en la región norte debido a una caída del 28 % en el canal mayorista. Sin embargo, el canal retail creció un 8 %, por lo que recomendamos reforzar la estrategia comercial en mayoristas."

Ese tipo de comunicación genera valor.

¿Por qué ocurre este error?

El mensaje clave es:

(fetch first)

y

the remote contains work that you do not have locally

Esto significa que tu repositorio en GitHub ya tiene un commit que tu computadora no tiene.

Lo más probable es que al crear el repositorio marcaste esta opción:

✅ Add a README

GitHub creó automáticamente un primer commit:

README.md

Mientras tanto, tú también hiciste un primer commit en tu computadora.

Ahora existen dos historiales distintos.

GitHub

│

└── Commit A (README)



Local

│

└── Commit B (Initial commit)

Git no sabe cuál conservar.

Verificar estado

Ejecuta:

git status

Luego:

git log --oneline

Y también:

git remote -v

Con eso normalmente ya puedo diagnosticar cualquier problema de Git.

Solución

La solución (recomendada)

Como apenas estamos empezando el curso, la mejor opción es traer el README de GitHub y combinar ambos historiales.

Ejecuta:

git pull origin main --allow-unrelated-histories

Probablemente Git abra un editor de texto para confirmar el mensaje del merge.

Si aparece Vim (pantalla negra), haz esto:

1\.	Presiona Esc 

2\.	Escribe: 

:wq

3\.	Presiona Enter. 

Si se abre el Bloc de notas, simplemente guarda y cierra.
